1. Overview

CareTrack is a staff-only clinic management and medical records application. Users sign in with seeded staff accounts, then work inside a protected dashboard where role determines which records can be viewed, created, updated, or deleted. The UI brand text currently appears as Medora Pro, while package metadata and backend messages still use CareTrack.

Current implementation: a React single-page app in frontend/src served by Vite and a CommonJS Express API in backend/src.
Persistence status: the project does not currently connect to PostgreSQL, Firebase, or another durable store. Records live in module-level arrays and reset when the backend process restarts.

Localization Switch UI strings between Uzbek and Russian, persisted in localStorage. LanguageContext.jsx, MainLayout.jsx

Frontend modules

  • services/api.js creates one Axios instance, attaches Authorization: Bearer <token>, and redirects to /login on non-auth 401 responses.
  • utils/auth.js stores, removes, and decodes JWTs from localStorage. Role checks are client-side convenience only; backend middleware is the source of truth.

  • i18n/LanguageContext.jsx stores selected language under caretrack.language and exposes translation helpers for role, gender, severity, and shift labels.

7. Data Model

Data is loaded from JavaScript files when the backend process starts. IDs are generated with uuid, passwords are hashed with bcrypt, and timestamps are ISO strings. The SQL file is retained as a historical relational schema, but db.js explicitly marks database connection code as deprecated.

User

{
  id: string,
  username: string,
  email: string,
  password: bcrypt_hash,
  role: "Admin" | "Clinician" | "Receptionist",
  created_at: ISO_datetime
}

Doctor

{
  id: string,
  user_id: string | null,
  name: string,
  specialty: string,
  department: string,
  contact: string | null,
  position: string,
  shift: "day" | "night" | "rotating",
  created_at: ISO_datetime
}

Patient

{
  id: string,
  first_name: string,
  last_name: string,
  age: number,
  gender: string,
  phone: string | null,
  address: string | null,
  doctor_id: string | null,
  created_at: ISO_datetime
}

Diagnosis

{
  id: string,
  patient_id: string,
  icd_code: string,
  description: string,
  severity: "mild" | "moderate" | "severe",
  created_at: ISO_datetime
}

Relationships

  • Doctor.user_id optionally links a doctor profile to a clinician user account.
  • Patient.doctor_id links a patient to one doctor. Doctor deletion sets this value to null.
  • Diagnosis.patient_id links a diagnosis to one patient. Patient deletion removes linked diagnoses.
  • Clinician scoping is derived by finding the doctor whose user_id matches req.user.id, then using that doctor's patient assignments.

8. Auth and Access

Authentication is username/password based. On login, the API signs a 24-hour JWT containing id, username, email, and role. The frontend stores the token in localStorage and sends it with API requests.

Role Backend permissions Frontend behavior
Admin Full doctor, patient, diagnosis, dashboard, and user-registration access. Can access user management, create doctors, export CSV, and delete records.
Clinician Can read assigned patients, read/create/update diagnoses, update patients, and view scoped dashboard stats. Sees clinician dashboard, diagnosis workflows, and patient directory for assigned data.
Receptionist Can read doctors and patients, create/update patients, and view dashboard stats. Cannot access diagnoses endpoints. Sees intake-focused dashboard, patient registration, and doctor roster.
Production hardening required: JWT_SECRET falls back to your-secret-key-change-in-production. A real deployment must provide a strong secret, avoid demo credentials, and move from localStorage token handling to a safer session strategy if the threat model requires it.

9. Core Workflows

Login workflow

  1. User submits username and password in LoginPage.
  2. authService.login posts to /api/auth/login.
  3. authController.login finds the user, compares bcrypt hash, signs JWT, and returns user summary.
  4. Frontend stores token with setToken and navigates to /.

Doctor creation workflow

  1. Admin opens /doctors/new.
  2. Form submits doctor details plus clinician username and password.
  3. API validates required fields, password length, and duplicate username.
  4. Controller creates a Clinician user and a linked doctor profile using user_id.

Patient profile workflow

  1. Frontend requests GET /api/patients/:id.
  2. API verifies the token; clinicians must own the assigned doctor relationship for that patient.
  3. Controller returns patient fields, doctor summary, and diagnosis history sorted newest first.

Dashboard workflow

  1. Dashboard calls GET /api/dashboard/stats.
  2. Admin and Receptionist receive global counts. Clinician receives counts derived from assigned patients only.
  3. Frontend renders role-specific stat cards, quick actions, recent tables, shift coverage, and severity chart.

Patients Required first name, last name, age, gender; doctor existence check on create. 400 validation failure, 403 clinician scope denial, 404 missing patient.
Diagnoses Required patient, ICD code, description, severity; severity enum; patient existence on create. 400 validation failure, 403 clinician scope denial, 404 missing diagnosis.

Frontend forms also use local validation utilities and translated error messages, but backend validation remains the authoritative enforcement point.

Recommended deployment shape

  1. Deploy backend as a Node.js service with a strong JWT_SECRET and explicit PORT.
  2. Deploy frontend static assets from frontend/dist to a static host.
  3. Set VITE_API_BASE_URL at build time to the public API URL.
  4. Add a durable database before using the system for real clinical records.
  5. Configure HTTPS, strict CORS origins, monitoring, backups, audit logging, and rate limiting.

13. Architecture Notes

Not production-ready for real medical records: data is in memory, demo credentials are visible in the UI, there is no audit trail, no durable backup, no encryption-at-rest story, no rate limiting, no request validation library, and no healthcare compliance controls.
Observation Impact Recommended next step
In-memory arrays are the active data store. All writes disappear after restart and cannot support multi-instance deployment. Implement a database layer, likely PostgreSQL based on the existing init.sql intent.
init.sql does not match the active doctor fields fully. Fields like user_id, position, and shift are missing from the old SQL schema. Update schema before database migration.
Client decodes JWT for route gating. Good UX, but not a security boundary. Keep backend checks mandatory and add API tests around every protected route.
Patient update route allows clinicians by role, but does not scope the target patient in update. A clinician may be able to update an unassigned patient if they know the ID. Add the same clinician ownership check used by getById.
Diagnosis create/update does not fully enforce clinician patient ownership. A clinician could create or move a diagnosis for an unassigned patient. Validate assigned-patient ownership on create and update.
CORS allows all origins by default. Acceptable for local development, too broad for production. Restrict CORS to approved frontend origins.
